## OOP là gì?

**Lập trình hướng đối tượng (Object-Oriented Programming — OOP)** là một paradigm tổ chức code xung quanh các *đối tượng* (object) thay vì các hàm và logic. Mỗi đối tượng gói chung **dữ liệu** (thuộc tính) và **hành vi** (phương thức) liên quan.

<Callout type="tip">
Một câu hỏi cực kỳ phổ biến: "Bạn hãy giải thích OOP qua một ví dụ thực tế." Hãy chuẩn bị 1-2 ví dụ ngắn gọn (ví dụ: lớp `BankAccount`).
</Callout>

## Bốn trụ cột

OOP có **4 trụ cột** mà bạn bắt buộc phải biết:

1. **Encapsulation** — Đóng gói dữ liệu và hành vi vào một đơn vị, ẩn chi tiết bên trong.
2. **Inheritance** — Kế thừa hành vi từ lớp cha.
3. **Polymorphism** — Một interface, nhiều hành vi.
4. **Abstraction** — Ẩn đi cài đặt phức tạp, chỉ phơi ra phần cần dùng.

## Ví dụ tối giản

```cpp
class BankAccount {
    std::string owner;  // private by default trong C++ class
    double balance;

public:
    BankAccount(const std::string& name, double initial)
        : owner(name), balance(initial) {}

    void deposit(double amount) {
        if (amount <= 0) throw std::invalid_argument("Amount must be positive");
        balance += amount;
    }

    double getBalance() const { return balance; }
    std::string getOwner() const { return owner; }
};
```

`balance` và `owner` được ẩn (encapsulation), người dùng chỉ tương tác qua `deposit` và `getBalance`.

## Class và Object

**Class** là bản thiết kế (blueprint) — chỉ tồn tại trong code, chưa chiếm bộ nhớ runtime.  
**Object** là thực thể cụ thể được tạo ra từ bản thiết kế, **chiếm bộ nhớ** trong RAM lúc chương trình chạy.

```cpp
// Class: bản thiết kế
class Car {
public:
    std::string color;
    int speed = 0;
    void accelerate(int n) { speed += n; }
};

// Objects: thực thể thật trong bộ nhớ, độc lập nhau
Car car1;             // object trên stack
car1.color = "Red";

Car* car2 = new Car();  // object trên heap — phải delete thủ công
car2->color = "Blue";
delete car2;

// Nhiều objects từ 1 class — hoàn toàn độc lập
Car a, b, c;  // thay đổi a không ảnh hưởng b hay c
```

## Vì sao interviewer hay hỏi?

- Đánh giá khả năng **thiết kế** chứ không chỉ viết code chạy.
- Kiểm tra hiểu biết **trade-off**: khi nào dùng kế thừa, khi nào dùng composition.
- Là nền tảng cho **design patterns** và **SOLID**.

## Tóm tắt

- OOP gom dữ liệu + hành vi vào object.
- Bốn trụ cột: encapsulation, inheritance, polymorphism, abstraction.
- Phỏng vấn thường yêu cầu giải thích kèm ví dụ thực tế.
